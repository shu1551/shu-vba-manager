# 変更履歴

## 2026-07-16

### 追加（予行演習run とブックの開閉 — コマンド数 72 → 75）

- **`rehearse`（別名: `予行演習`）** — マクロを**本体の完全コピーに試し撃ちして、何が変わるかを差分で報告**する非破壊run。流れ: `SaveCopyAs`（未保存の変更込みの「今この瞬間」）→ 別インスタンス（DispatchEx・非表示）でコピーを開く（Workbook_Open は起こさない・健診モードと同じ思想）→ 実行前 snapshot → マクロ実行（ダイアログ安全解除は run-macro と同じく常設・`--auto-dialog` も可）→ 実行後 snapshot → `snapshot-diff` で事実報告。マクロが途中で落ちても、そこまでの変化を差分で報告する（それも予行演習の収穫）。結果コピーと前後 snapshot は検分できる形で残す（`--discard` で削除、`--out` で保存先指定、`--addins` でアドイン入り環境）。**本体に焼くかどうかは、この報告を見て人が決める**
  - ※ 非破壊が保証できるのは**ブックの中だけ**。ファイル出力・メール送信など「外への副作用」を持つマクロまでは止められない
- **`open`** — ブックを「人が開くのと同じ場所」に開く。既に開いていれば前面化のみ（二重に開かない）／見えている Excel が居ればそこに合流／Excel 未起動なら通常起動（COM 起動と違い、アドイン・PERSONAL.XLSB が普段どおり読み込まれる）。**非表示の残骸 Excel しか居ないときは開かずに止まる**（残骸に取り込まれると「開いたのに見えない・アドインが効かない」異常環境になるため）
- **`close`** — 開いているブックを**1冊だけ**閉じる。鎧の三点セット: ①ブック名指し必須 ②保存方針（`--save` / `--no-save`）の明示必須 ③確認プロンプト（`-y` でスキップ）。Excel 本体は終了せず、PERSONAL.XLSB とアドインブックは閉じない。同名ブックが複数開いているときはフルパスでの名指しを要求する
- 内部: `snapshot` の本体を `_snapshot_doc()`（Workbook を直接受け取る部品）として切り出し、`snapshot` コマンドと `rehearse` で共用

### テスト

## 2026-07-14

### 修正（最重要: 名前衝突ガード自身が連番モジュールを生んでいた）

- **Import 成功後の保存失敗で、ツールが自分で連番モジュールを作る事故を根絶** — Remove+Import 系3経路（`replace-procedure` の Attribute 経路 / `replace-module` / `reorder-macro`）の例外処理が `removed` フラグしか見ておらず、**Import が成功した後の `wb.Save()` が失敗すると「モジュールが消えた」と誤認してバックアップを重ね Import** していた。期待名のモジュールは既に正しく存在するため必ず VB_Name が衝突し、旧コード入りの連番モジュール（例: `shu003` の隣に `shu0031`）が生まれる。しかも消滅待ち15秒＋改名待ち20秒を空振りしてから失敗を告げるため、35秒待たされた末に汚染だけが残っていた。`imported` フラグを導入し、**バックアップ復旧は「Import 自体が失敗したとき」だけ**に限定。実機の VBE で連番化を再現して確認済み
- **誤誘導していた案内文を訂正** — 従来は衝突時に「旧モジュールが消えているのを確認してから連番側を改名せよ」と案内していたが、この筋では旧名のモジュールこそが正しく置換された本体。指示に従うと**正しい方を潰す**。「取り込みは成功している。壊れているのは保存だけ。再 Import はするな」と正しく伝えるようにした
- **`_save_with_retry` を追加** — Excel はセル編集中・メニュー展開中・モーダル表示中に COM 呼び出しを拒否する（`RPC_E_CALL_REJECTED` 等）。Import 直後の保存だけはこれで諦めず、少し待って撃ち直す。恒久的な失敗（読み取り専用・ロック）はそのまま送出
- **`optimize_vba_modules.py`** — `Remove` 直後に素の `VBComponents.Import` を撃つ無防備な経路だった（2026-07-11 の shu005 消滅事故と完全に同型）。本体と同じ実名検証つき Import に委譲し、置換前のバックアップ・復旧・非ゼロ終了を追加。例外を握ったあとでも「完了」と表示していたのも修正
- **`live_sync_vba.py`** — `DeleteLines` でモジュールを空にしてから `AddFromString` する作りで、**注入に失敗すると空のまま保存される**うえ、全モジュールの更新に失敗しても `Successfully saved changes` と報告していた。失敗時に旧コードを書き戻すロールバック、1件でも失敗したら保存しない・非ゼロ終了を追加。Excel 未起動時に新インスタンスを作ってゾンビを残すフォールバックも撤去
- **`delete-module`** — `Remove` の完了を待たずに保存して「削除完了」と報告していた。VBE の Remove は遅延完了することがあり、ファイルには残ったまま成功と告げる食い違いが起きる。消滅を確認してから保存するようにした
- **`form_tool.py` の `copy-form`** — `.frm` の `Begin` 行はフォーム名の後ろに空白が付くため、末尾トークンだけを書き換えるガードが `rpartition` で空文字を掴み、**一度も発火していなかった**（GUID 破壊を防ぐはずの防御が死んでいた）。末尾空白を考慮して比較・復元するよう修正

### 追加（テストが「何度もコケる」の構造的な根治）

- **`test_e2e_com.py` / `conftest.py`（新規）** — 実機の Excel/VBE を使う E2E テストを正式ファイルに昇格。従来は使い捨てスクリプトとして毎回書き直され、同じ穴を踏み直していた。既定ではスキップ（オプトイン）
  - 普段: `py -m pytest -q` → 純ロジックのみ・約1.5秒
  - 実機: `py -m pytest test_e2e_com.py --run-e2e -q`
  - **Excel の死は非同期**（`Quit` から実測2〜3秒）。死にきる前に次の操作を始めると `GetActiveObject` が「まだ死んでいない Excel」に相乗りして挙動が変わる——これがテストが実行ごとにコケたり通ったりする正体だった。各テストの入口で「無人」を確認してから進むようにして構造的に解消（3回連続実行で全通過を確認）
  - 作業中の Excel が起動していたら理由を名指しして skip する（開いているブックを巻き込まない）。ゾンビ Excel を残したら失敗させる
- **回帰テストを構文木で機械照合** — 「バックアップの再 Import が `imported` フラグで守られているか」を AST で検査。将来ガードが外されたら落ちる
